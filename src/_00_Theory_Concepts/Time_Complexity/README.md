# ⏳ 시간 복잡도 (Time Complexity) 판독 가이드

## 1. 핵심 법칙: "1초 = 1억 번($10^8$)"
- 코딩 테스트의 일반적인 시간 제한은 1초입니다.
- 나의 알고리즘 연산 횟수가 1억 번을 넘지 않도록 설계해야 합니다.

## 2. $N$의 크기에 따른 알고리즘 선택 공식

| $N$의 범위 | 허용 시간 복잡도 | 추천 알고리즘 |
| :--- | :--- | :--- |
| $N \le 20$ | $O(2^N)$ | 백트래킹, 완전 탐색 |
| $N \le 500$ | $O(N^3)$ | 3중 for문 (예: 블랙잭) |
| $N \le 2,000$ | $O(N^2)$ | 2중 for문 (단순 브루트포스) |
| **$N \le 100,000$** | **$O(N \log N)$** | **정렬, 이분 탐색 (오늘 푼 수 찾기)** |
| **$N \le 10,000,000$** | **$O(N)$** | **1중 for문, 투 포인터 (내일 풀 문제)** |
| 그 이상 | $O(\log N)$ | 이분 탐색 (Binary Search) |



## 3. 알고리즘별 '가성비' (Big-O)
- **$O(1)$**: 해시 테이블 탐색 (`HashSet.contains`) - *가장 빠름*
- **$O(\log N)$**: 이분 탐색 - *데이터가 커도 끄떡없음*
- **$O(N)$**: 단순 for문, 투 포인터 - *데이터 수만큼 정직하게 실행*
- **$O(N \log N)$**: 표준 정렬 (`Arrays.sort`) - *코테의 마지노선*
- **$O(N^2)$**: 이중 for문 - *데이터가 1만 개만 넘어도 위험*

---
**기억할 점**: 문제를 읽자마자 $N$의 범위를 확인하고, 어떤 '도구(알고리즘)'를 꺼낼지 결정할 것!
