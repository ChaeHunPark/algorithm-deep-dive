## 🚀 Algorithm Deep-Dive: Level 10 (String)
### 📌 백준 5052 - 전화번호 목록

#### 1. 문제 분석 및 접근
* **문제 유형**: 문자열 탐색 및 접두사 판별 (Trie 자료구조)
* **핵심 이슈**: $N$개의 전화번호 중 어느 하나가 다른 번호의 **접두어(Prefix)**가 되는지 여부를 $O(N \times L)$ 시간 내에 판별해야 함.
* **자료구조 선택**: 일반적인 문자열 비교($O(N^2)$)는 시간 초과 가능성이 큼. 문자열의 각 문자를 노드로 구성하여 공통 접두사를 공유하는 **트라이(Trie)**가 최적임.

#### 2. 핵심 설계 지점
* **TrieNode 구조**: `Map<Character, TrieNode>`를 사용하여 자식 노드를 동적으로 관리하고, `isEndOfWord` 플래그로 단어의 종착역을 표시함.
* **Java Java-ism**: `computeIfAbsent` 메서드를 활용하여 "노드가 없으면 생성하고, 있으면 가져오는" 로직을 한 줄로 간결하게 구현.
* **일관성 체크 로직 (Dual-Check)**:
    1.  **삽입 중**: 새로운 번호를 삽입하는 경로에 이미 `isEndOfWord == true`인 노드가 있다면, 이전의 짧은 번호가 나의 접두어임.
    2.  **삽입 후**: 번호 삽입이 끝났는데 해당 노드의 `childNodes`가 비어있지 않다면, 내가 다른 긴 번호의 접두어임.

#### 3. 예리한 디버깅 포인트 (Self-Feedback)
* **정렬의 힘**: 입력받은 문자열들을 사전순(`Arrays.sort()`)으로 정렬하면, 접두어 관계인 번호들이 항상 인접하게 됨. 이를 통해 삽입 로직에서 "내가 삽입하려는데 이미 끝난 단어가 있는 경우"만 체크해도 완벽한 판별이 가능해짐.
* **초기화 주의**: 테스트 케이스(`T`)가 여러 개인 문제이므로, 매 케이스마다 `Trie` 객체를 새로 생성하여 이전 데이터와의 간섭을 원천 차단함.
* **메모리 관리**: `HashMap`은 유연하지만 메모리 사용량이 많음. 숫자만 다루는 이번 문제에서는 다른 자료구조 딥다이브 레포지토리에서 구현한 `TrieNode[10]` 배열을 쓰는 것이 더 예리한 최적화가 될 수 있음.
