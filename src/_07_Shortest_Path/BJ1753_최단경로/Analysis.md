# 🚀 1753번: 최단경로 (Dijkstra) 분석

## 1. 문제 개요
* **핵심 목표**: 방향 그래프에서 주어진 시작점에서 다른 모든 정점으로의 최단 경로 구하기.
* **사용 알고리즘**: 다익스트라 (Dijkstra)
* **주요 제약**: 정점(V) 2만 개, 간선(E) 30만 개. 성능 최적화를 위해 $O(E \log V)$ 방식 필수.

---

## 2. 알고리즘 설계 및 구현 (My Logic)

### 🏗️ 자료구조 선택
1. **인접 리스트 (`List<Node>[]`)**: 2차원 배열은 메모리 초과 위험이 커서 사용. 각 정점마다 연결된 간선 정보만 저장하여 효율성 극대화.
2. **최단 거리 배열 (`dist[]`)**: 시작점으로부터의 거리를 기록. 초기값은 무한대(`INF`)로 설정.
3. **우선순위 큐 (`PriorityQueue`)**: 가중치가 가장 낮은 노드부터 꺼내어 탐색 횟수를 줄임.

### ⚙️ 핵심 프로세스 (3단계)
1. **초기화**: 시작점의 거리는 `0`, 나머지는 `INF`. 시작 노드를 PQ에 삽입.
2. **가까운 노드 탐색**: PQ에서 거리가 가장 짧은 노드를 꺼냄. (이때 이미 처리된 더 짧은 경로가 있다면 무시)
3. **완화 (Relaxation)**:
    - 현재 노드를 거쳐서 다음 노드로 가는 거리($dist[cur] + weight$) 계산.
    - 기존에 알고 있던 거리($dist[next]$)보다 짧다면 **메모장 업데이트** 후 PQ에 삽입.

---

## 3. 핵심 인사이트 (Trouble Shooting)

### ✅ BFS vs 다익스트라
* **BFS**: 가중치가 모두 1일 때 사용. `visited` 배열로 한 번 방문하면 끝.
* **다익스트라**: 가중치가 다를 때 사용. 방문했더라도 **더 짧은 거리가 나오면 계속 갱신(Update)**. 즉, "BFS의 탐색 구조 + DP의 최적화 원리"가 결합된 형태.

### ✅ 구현 시 주의사항
* **오버플로우**: `Integer.MAX_VALUE` 대신 적당히 큰 값(`100,000,000`)을 사용하여 덧셈 연산 시 오류 방지.
* **성능**: 대량의 정점 출력을 위해 `StringBuilder`를 사용하여 시간 초과 방지.

## 4. 회고
처음에는 복잡한 자료구조와 조건문 때문에 어렵게 느껴졌지만, 직접 손으로 데이터를 트래킹해보며 **"가장 짧은 곳부터 방문하며 지도를 최신화한다"**는 단순한 논리를 체득했다. 다익스트라는 결국 **'지능적으로 길을 찾는 내비게이션'**과 같다는 것을 이해함.
