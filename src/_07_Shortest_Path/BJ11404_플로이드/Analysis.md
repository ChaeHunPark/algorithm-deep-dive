# 🚌 11404번: 플로이드 (Floyd-Warshall) 분석

## 1. 문제 개요
* **핵심 목표**: 모든 도시 쌍 (i, j)에 대하여 도시 i에서 j로 가는 최소 비용을 계산하는 전경로 최단 거리 문제.
* **사용 알고리즘**: 플로이드-워셜 (Floyd-Warshall) 알고리즘.
* **기술적 근거**: 정점의 개수 $N$이 100으로 매우 적으며, 모든 정점 사이의 최단 거리를 한 번에 구해야 하므로 $O(N^3)$의 복잡도를 가지는 플로이드-워셜이 가장 효율적임.

---

## 2. 알고리즘 설계 및 구현 (My Logic)

### 🏗️ 데이터 구조
1. **2차원 거리 배열 (`int[][] dist`)**: 모든 노드 쌍 간의 최단 거리를 저장하는 메모이자 테이블 (DP Table). 초기값은 `INF`로 설정하며, 자기 자신으로 가는 경로는 `0`으로 초기화.
2. **인접 행렬 기반 입력**: 도시 간 버스 노선 정보를 입력받아 `dist` 배열에 직접 기록.

### ⚙️ 핵심 프로세스
1. **초기화 및 입력**: 버스 정보를 입력받아 저장하되, 동일 노선에 여러 버스가 있을 수 있으므로 `Math.min()`으로 최소 비용만 유지.
2. **3중 루프 실행 (K-I-J)**:
    - **가장 바깥 루프 (K)**: 경유할 정점을 선택.
    - **중간 루프 (I)**: 출발 정점을 선택.
    - **안쪽 루프 (J)**: 도착 정점을 선택.
    - `dist[i][j] > dist[i][k] + dist[k][j]`일 경우 데이터를 갱신 (Relaxation).
3. **결과 출력**: 계산이 끝난 후 여전히 `INF`인 지점은 갈 수 없는 경로이므로 `0`으로 변환하여 출력.

---

## 3. 핵심 인사이트 & 시행착오 (Trouble Shooting)

### ✅ K-I-J 변수명과 루프 순서의 관습
* 처음에는 왜 루프 순서가 `i, j, k`가 아닌 `k, i, j`여야 하는지 의문이었으나, 두 가지 측면에서 명확한 이유를 찾음:
    1. **논리적 필수성**: 가장 바깥 루프인 **경유지(K)**가 고정되어야만 이전 단계의 최단 거리 데이터를 기반으로 다음 단계를 빌드업하는 DP의 무결성이 보장됨.
    2. **관용적 약속**: 학계와 실무에서 **K**는 'K-node' 혹은 'K-point(경유점)'를, **I**는 'In(출발)', **J**는 'Junction/Join(연결점/도착)'을 의미하는 관습이 있음. 이 규칙을 따르면 복잡한 3중 루프에서도 인덱스 실수를 줄이고 가독성을 극대화할 수 있음을 깨달음.
### ✅ 2차원 배열 버전의 DP (Memoization)
* 별도의 DP 테이블을 선언하지 않아도 `dist[i][j]` 자체가 메모이전 공간 역할을 수행함. **"이미 저장된 최적의 경로(`dist[i][k]`)를 활용해 미래의 경로를 결정한다"**는 DP의 본질적인 원리를 그래프 알고리즘에서 확인한 것이 큰 수확이었음.

### ✅ 시간 복잡도와 제약 사항
* $N=100$일 때 $N^3$은 $1,000,000$으로, 자바의 시간 제한 내에서 매우 안정적으로 동작함. 하지만 $N$이 커질 경우(예: $N=1,000$ 이상) 다익스트라를 N번 돌리는 방식이 더 유리할 수 있다는 점을 인지함.

---

## 4. 회고
> 처음 접했을 때는 3중 `for`문이 단순히 무식하게 모든 경로를 찾는 브루트포스처럼 느껴졌으나, 실상은 **과거의 최적해를 재사용하여 현재의 상태를 업데이트하는 정교한 DP**임을 깨달았다.
>
> 처음이라 데이터가 갱신되는 흐름을 추적하는 과정이 다소 어려웠지만, **데이터 무결성을 유지하며 상태를 점진적으로 최적화**하는 감각을 익힐 수 있었던 유익한 시간이었다. 특히 `INF` 설정 시 오버플로우를 방지하면서도 경로 없음을 명확히 구분하는 디테일의 중요성을 다시 한번 상기함.
