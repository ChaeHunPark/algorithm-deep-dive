# 🚌 1717번: 집합의 표현 (Union-Find) 분석

## 1. 문제 개요
* **핵심 목표**: 초기 상태의 $n+1$개 집합에 대해 합집합 연산(`0`)과 두 원소의 동일 집합 여부 확인 연산(`1`)을 효율적으로 수행하기.
* **사용 알고리즘**: 유니온 파인드 (Union-Find / Disjoint-Set) 알고리즘.
* **기술적 근거**: 원소 개수 $N=1,000,000$, 연산 횟수 $M=100,000$으로 매우 방대함. 일반적인 그래프 탐색으로는 시간 초과가 발생하므로, 거의 상수 시간에 가까운 성능을 내는 유니온 파인드가 최적임.

---

## 2. 알고리즘 설계 및 구현 (My Logic)

### 🏗️ 데이터 구조
1. **부모 배열 (`int[] parent`)**: 각 노드의 부모 정보를 저장하는 테이블. 초기값은 `parent[i] = i`로 설정하여 모든 노드가 자기 자신을 대표하도록 초기화.
2. **트리 구조 기반 집합**: 그룹을 트리 형태로 관리하며, 각 트리의 루트(Root) 노드를 해당 집합의 대표값(대빵)으로 간주함.

### ⚙️ 핵심 프로세스
1. **Find 연산 (경로 압축 적용)**:
    - 재귀적으로 부모를 타고 올라가 루트 노드를 찾음.
    - 리턴 과정에서 `parent[x] = find(parent[x])`를 통해 경로상의 모든 노드가 루트를 직접 가리키도록 업데이트 (Path Compression).
2. **Union 연산 (집합 병합)**:
    - 두 원소 $a, b$의 루트를 각각 찾음.
    - 두 루트가 다를 경우, 한쪽 루트의 부모를 다른 쪽 루트로 설정하여 두 그룹을 하나로 통합.
3. **조건부 처리**:
    - `type 0`: `union(a, b)` 수행.
    - `type 1`: `find(a) == find(b)`를 확인하여 "YES" 또는 "NO" 출력.

## 3. 핵심 인사이트 & 시행착오 (Trouble Shooting)

### ✅ 단순 부모 연결 vs 루트 연결 (Data Integrity)
* 단순히 `parent[a] = b`로 연결할 경우, 그룹 전체가 아닌 개별 노드만 이동하게 되어 데이터 무결성이 깨짐. 반드시 `find()`를 통해 **각 그룹의 대빵(Root)**을 찾아 연결해야 가문 전체가 올바르게 통합됨을 이해함.

### ✅ 경로 압축(Path Compression)의 자가 치유 능력
* 입력 순서에 따라 트리가 일렬로 길어지는 **편향 트리(Skewed Tree)**가 발생할 수 있으나, `find` 연산 시 부모를 루트로 갱신해주는 한 줄의 코드로 트리를 납작하게(Flat) 만듦. 이는 일종의 **"조회 시 최적화(On-the-fly Optimization)"** 전략으로, 이후 연산 속도를 비약적으로 높여줌.

### ✅ 예외 케이스 처리 (`0 1 1`)
* 자기 자신을 합치는 연산(`union(a, a)`)이 들어와도 `if (rootX != rootY)` 조건을 통해 불필요한 수정을 방지함. 또한 방대한 입출력 처리를 위해 `Scanner` 대신 `BufferedReader`와 `StringBuilder`를 사용하여 시간 초과 리스크를 관리함.
