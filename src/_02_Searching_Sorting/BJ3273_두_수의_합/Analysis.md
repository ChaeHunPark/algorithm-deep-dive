# 📂 Algorithm Analysis: BJ3273 (두 수의 합)

## 1. 문제 요약
- **핵심**: n개의 서로 다른 정수 중 합이 x가 되는 쌍의 개수 구하기.
- **제한 사항**: n은 최대 100,000 / 시간 제한 1초.

## 2. 시간 복잡도 설계 (핵심!)

### ❌ 실패한 접근: 브루트 포스 (O(N²))
- **방식**: 2중 for문을 돌며 모든 쌍을 확인.
- **연산 횟수**: $100,000 \times 100,000 = 10,000,000,000$ (**100억 번**)
- **결과**: 시간 초과 (1초 제한을 100배 초과)

### ✅ 성공한 접근: 정렬 + 투 포인터 (O(N log N))
- **방식**: `Arrays.sort()` 후 양 끝에서 포인터를 좁혀가며 탐색.
- **연산 횟수**:
    1. 정렬: $100,000 \times \log_2 100,000 \approx$ **170만 번**
    2. 투 포인터 탐색: 최대 **10만 번**
    3. 총합: **약 180만 번**
- **결과**: **0.02초** 내외로 통과 (매우 효율적)



## 3. 투 포인터 이동 원리
- **sum == target**: 쌍을 찾았으므로 `result++`. 현재 `left`와 `right`는 유일한 짝궁이므로 둘 다 안쪽으로 이동(`left++`, `right--`).
- **sum < target**: 합이 작으므로 더 큰 숫자가 필요함. 오름차순 정렬 상태이므로 `left++`.
- **sum > target**: 합이 크므로 더 작은 숫자가 필요함. `right--`.

## 4. 새롭게 배운 점
- **N의 규모**를 보고 알고리즘을 역추적하는 법을 익힘. ($N=10^5$ 일 때 $O(N^2)$은 금기사항!)
- 로그($\log$)가 왜 대용량 데이터 처리의 마법인지 체감함 ($10^5$이 고작 $17$이 되는 기적).
- "게으른 완벽주의자"로서, 무작정 코딩하기보다 **연산 횟수를 먼저 계산하는 습관**이 삽질을 줄여준다는 것을 깨달음.
