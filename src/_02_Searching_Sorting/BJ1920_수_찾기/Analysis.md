# 🔍 알고리즘 분석: 수 찾기 (Binary Search)

## 1. 문제 개요
- **목표**: $N$개의 정수 배열 $A$에 $M$개의 특정 숫자들이 존재하는지 확인.
- **제약 조건**:
    - $N, M \le 100,000$
    - 시간 제한: 1.0초 (약 $10^8$ 연산)

---

## 2. 복잡도 분석 (Complexity Analysis)

### A. 브루트 포스 ($O(N \times M)$)
- **방법**: 질문 하나당 배열 전체를 순차 탐색.
- **계산**: $100,000 \times 100,000 = 10,000,000,000$ (100억 번).
- **결과**: **시간 초과(TLE)**. 1초 내에 수행 불가능.

### B. 이분 탐색 ($O(M \log N)$) - *선택한 방식*
- **방법**: 배열 $A$를 정렬($O(N \log N)$)한 뒤, 각 질문에 대해 이분 탐색($O(\log N)$) 수행.
- **계산**: $10^5 \log 10^5 \approx 10^5 \times 17 \approx 1.7 \times 10^6$ 번 연산.
- **결과**: **성능 최적화 완료**. 1초 내에 매우 여유롭게 통과.

### C. 해시 탐색 ($O(N + M)$)
- **방법**: `HashSet`에 데이터를 담고 $O(1)$으로 존재 여부 확인.
- **특징**: 가장 빠르지만 메모리 사용량이 배열 대비 높음.

---

## 3. 핵심 로직 및 구현
- **정렬 필수**: `Arrays.sort()`를 통해 이분 탐색의 전제 조건인 오름차순 정렬 수행.
- **탐색 효율**: 직접 `while`문을 구현할 수도 있으나, 자바 표준 라이브러리인 `Arrays.binarySearch()`를 활용하여 안정성 확보.
- **I/O 최적화**: 데이터 양이 많으므로 `Scanner` 대신 `BufferedReader`와 `StringBuilder`를 사용하여 입출력 병목 현상 방지.

---

## 4. 결론
데이터의 규모가 커질수록 $O(N^2)$ 이상의 알고리즘은 지양해야 하며, 정렬된 상태를 활용할 수 있는 **이분 탐색**이 매우 효율적인 대안임을 확인하였다.
