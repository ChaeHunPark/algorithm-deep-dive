# 📂 Algorithm Analysis: BJ2805 (나무 자르기)

## 1. 문제 요약
- **핵심**: 절단기 높이 $H$를 설정하여, 잘려나간 나무 길이의 합이 적어도 $M$ 이상이 되는 $H$의 최댓값 찾기.
- **제한 사항**: 나무 수 $N \le 1,000,000$, 목표 길이 $M \le 2,000,000,000$, 나무 높이 $\le 2,000,000,000$.

## 2. 시간 복잡도 설계 (Parametric Search)

### ❌ 실패하는 접근: 브루트 포스 ($O(H \times N)$)
- **방식**: 높이 $H$를 1부터 20억까지 하나씩 대입하며 확인.
- **연산 횟수**: $2,000,000,000 \times 1,000,000 = 2,000,000,000,000,000$ (2,000조 번)
- **결과**: 시간 초과 (지구가 멸망해도 안 끝남)

### ✅ 성공하는 접근: 매개 변수 탐색 ($O(N \log H)$)
- **방식**: 정답이 될 수 있는 높이 범위(0 ~ 20억)를 이분 탐색으로 좁혀나감.
- **연산 횟수**:
    1. 이분 탐색 횟수: $\log_2(2,000,000,000) \approx 31$번
    2. 각 탐색 시 나무 합산: $N = 1,000,000$번
    3. 총합: **약 3,100만 번**
- **결과**: **1초 내 통과 가능** (매우 효율적)



## 3. 핵심 로직: 결정 문제(Decision Problem)
- **질문**: "절단기 높이를 `mid`로 설정했을 때, 얻는 나무 합(`sum`)이 $M$ 이상인가?"
- **판단 (반비례 관계)**:
    - **YES (`sum >= M`)**: 나무가 충분함. 절단기를 더 **높여서** 나무를 아껴보자! → `low = mid + 1` (결과값 저장)
    - **NO (`sum < M`)**: 나무가 모자람. 절단기를 더 **낮춰서** 더 많이 자르자! → `high = mid - 1`

## 4. 백엔드 개발자의 디테일 (Troubleshooting)
- **Long 타입 사용**: 나무 합산 시 최대 2,000조까지 발생할 수 있으므로 `int` 대신 반드시 `long`을 사용해야 함 (Overflow 방지).
- **정렬 불필요**: 이분 탐색의 대상이 배열 인덱스가 아닌 '값(높이)'이므로 나무 배열을 정렬할 필요가 없음 ($O(N \log N)$ 비용 절감).

## 5. 새롭게 배운 점
- **매개 변수 탐색**: 최적화 문제를 "Yes/No" 결정 문제로 바꾸어 푸는 강력한 기법을 익힘.
- **데이터 범위의 직관**: 20억이라는 숫자를 보고 $O(\log H)$ 알고리즘을 떠올리는 감각을 기름.
