# 📑 [Algorithm Analysis] 백준 1931 - 회의실 배정

### 1. 문제 핵심 요약
* **목표**: 한 개의 회의실에서 열 수 있는 회의의 최대 개수 구하기.
* **핵심 조건**: 회의는 중단될 수 없으며, 끝남과 동시에 다음 회의가 시작될 수 있음.

---

### 2. 논리적 추론 (Reasoning)
* **왜 Greedy인가?**
    * 최대한 많은 회의를 넣으려면 **"가장 빨리 끝나는 회의"**를 우선 선택해야 함.
    * 종료 시간이 빠를수록 이후에 남는 시간이 많아져 다른 회의를 선택할 기회가 극대화됨.
* **정렬의 이중 조건**:
    1. **종료 시간 오름차순**: 탐욕적 선택의 핵심 기준.
    2. **시작 시간 오름차순**: 종료 시간이 같을 때 시작 시간이 빠른 것을 먼저 선택해야 `(2, 2)` 같은 즉시 종료 회의를 누락 없이 카운트 가능.
* **시간 복잡도**:
    * 정렬에 $O(N \log N)$, 탐색에 $O(N)$이 소요되므로 전체 **$O(N \log N)$**.

---

### 3. 코드 설계 (Implementation)
1. **데이터 구조**: `int[N][2]` 2차원 배열을 사용하여 시작/종료 시간 관리.
2. **Custom Sorting**: `Arrays.sort()`와 람다식을 사용하여 이중 정렬 조건 구현 (`o1[1] - o2[1]`).
3. **핵심 로직**:
    * `prev_end_time`: 직전 회의가 끝난 시간을 저장하는 기준점.
    * `if (time[i][0] >= prev_end_time)` : 현재 회의 시작 시간이 기준점 이후라면 선택(`count++`) 및 기준점 갱신.

---

### 4. Self-Correction & 핵심 포인트
* **자료구조 선택**: Key 중복 문제와 정렬 효율성을 고려하여 `HashMap` 대신 **배열(Array)**을 선택하여 메모리와 성능 최적화.
* **Modularization**: `sort()`와 `solved()` 메서드를 분리하여 '관심사의 분리'를 실천하고 가독성을 높임.
* **엣지 케이스**: 시작 시간과 종료 시간이 같은 회의`(2, 2)`를 처리하기 위해 '끝나는 시간'이 같을 때 '시작 시간'으로 2차 정렬하는 디테일을 챙김.

---

### 5. 한 줄 평
